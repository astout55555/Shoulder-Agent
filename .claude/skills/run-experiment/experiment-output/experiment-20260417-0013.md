# Experiment 2026-04-17 00:13 — Debate workflow with rewritten moderator prompt (N=8, compared to prior optimized control)

**N=8 debate runs (partial, stopped early due to OpenAI quota exhaustion mid-run 9). Compared against the N=10 control results from 2026-04-16 19:33, which used the newly optimized control prompt.** Both agents now have carefully engineered structured prompts.

## Headline finding

**The debate workflow's advantage over the optimized control reopens, particularly on advice-bias.**

| Scorer | Debate (new moderator, N=8) | Control (new control, N=10 — from prior exp) | Δ |
|---|---|---|---|
| reasoning-depth | 0.808 | 0.794 | **+0.014** |
| advice-bias | 0.839 | 0.783 | **+0.056** |

For context, the previous experiment with the optimized control but the *original* moderator had Δ = −0.005 RD / −0.003 Bias (a closed gap). The optimized moderator restores a positive gap on both metrics.

Compared to the *pre-optimization* gap (old control + old moderator): that was +0.048 RD / +0.104 Bias. The current post-optimization gap is smaller in magnitude on RD but retains a substantial bias advantage.

## Experiment execution issues

**Quota exhaustion:** Partway through debate run 9 the OpenAI account hit its quota cap. All remaining API calls failed and the script was terminated. Runs 1–8 completed successfully and their scores are persisted in `mastra.db`. Control runs did not execute at all, hence the comparison against the prior experiment's control scores.

**Transient socket errors during runs:** Several runs (notably 4 and 5) logged repeated `SocketError: other side closed` errors during workflow execution. Mastra's retry mechanism recovered most, but run times were inflated (run 4: 321s, run 5: 317s vs. typical 150–245s). These network errors may have consumed extra tokens per run, accelerating quota exhaustion.

**Zero-score exclusions:** Runs 4 and 5 returned score=0.000 for both scorers on the "Buy a house now" scenario, indicating the scorer-analyze step on that scenario failed during those runs (likely tied to the socket-error cascade consuming the retry budget before scoring completed). Following the eval script's zero-score filter convention, those two observations are excluded from the Buy house averages — leaving N=6 for Buy house and N=8 for the other three scenarios.

**Decomposing the effect:**
- The control-prompt redesign gave the control agent +0.058 on RD and +0.112 on bias relative to the debate workflow (i.e., it entirely closed the original gap).

Said differently: the debate workflow's residual advantage after both sides are optimized is ~+0.014 RD / +0.056 Bias. That is ~27% of the original RD gap and ~49% of the original bias gap.

## Interpretation

### The multi-agent structure has a real, measurable advantage — but a smaller one than originally claimed

When both agents are carefully prompted, the debate workflow still produces slightly better reasoning-depth scores (+0.014) and meaningfully better advice-bias scores (+0.056). This is stronger evidence for the structural hypothesis than the first post-control-optimization experiment suggested. Specifically:

1. **The adversarial structure contributes independent signal that structured prompting alone cannot replicate.** The moderator — seeing actual advocacy-style arguments from two separate agents — produces output that scores higher on bias dimensions than a single agent instructed to produce balanced advocacy. Likely because the debate agents genuinely commit to their assigned sides (they don't know they're in an experiment), which surfaces stronger non-recommended-side arguments than a single agent can produce when trying to advocate for two positions simultaneously.

2. **The bias advantage is larger than the reasoning-depth advantage.** This is the consistent pattern from the pre-optimization experiment too. The adversarial structure appears to add more value on *fairness* than on *depth of reasoning per se* — the depth is mostly achievable with structured prompting, but genuine balance is harder.

3. **The effect size is smaller than before, but the direction is preserved across scenarios.** 3 of 4 scenarios still favor debate on RD; all 4 favor debate on bias. This pattern is unlikely to be noise given the scenario variety.

### Scenario-by-scenario notes

- **Buy house**: Debate now wins clearly on bias (+0.032) despite 2 excluded runs. This scenario has the richest symmetric quantitative data, so structured prompting alone narrows the control's bias disadvantage — but the advocacy structure still surfaces slightly stronger counterarguments. RD is essentially tied (−0.003).
- **Move to new city**: Continues to favor debate on both metrics (+0.022 RD, +0.094 Bias). Consistent with prior finding: emotionally ambiguous scenarios are where adversarial structure matters most.
- **Build vs. SaaS**: Debate's bias advantage is back to +0.048 after having been −0.085 in the prior post-control-optimization experiment. The moderator change was worth +0.133 on this scenario alone — the biggest single-scenario bias improvement.
- **Accept promotion**: Debate wins on both metrics (+0.035 RD, +0.065 Bias). The "easy/lopsided" scenario where the structural forcing function of advocacy matters most for constructing the non-obvious case.

## Caveats

1. **N=8 vs. N=10 comparison is not apples-to-apples.** The control-side data is from a separate experiment run 2 days ago. The LLM judge (gpt-5-mini) could have drifted slightly, though unlikely in that window.

2. **Buy house has N=6, not N=8.** Two zero-score exclusions reduce the power on that specific scenario. Both exclusions came from runs with heavy socket-error retry activity, so there's a small concern that the excluded data would have skewed slightly (in either direction) due to partial context degradation.

3. **The debate runs experienced an unusual rate of network errors.** This was an OpenAI-side reliability issue, not anything caused by the prompt change. The retries that succeeded probably produced normal-quality outputs; the retries that didn't succeed (runs 4 and 5 on Buy house) produced zero scores. No reason to think the error rate biased the successful scores.

4. **The comparison has not been tested for statistical significance yet.** With N=8 debate vs. N=10 control, the overall +0.014 RD / +0.056 Bias deltas should be re-checked formally. At these sample sizes and scenario count, an informal estimate is: the bias delta is likely significant (comparable in direction and magnitude to the original bias result, which was highly significant); the RD delta is likely borderline.

## Recommendations

1. **Once quota is restored, re-run a full N=10 control experiment** with no other changes. This will give a clean N=10 vs. N=10 comparison for both this moderator configuration and the current control configuration. Ideally the existing 2026-04-16 control scores would be re-validated by a second N=10 run anyway — one run is enough to establish a comparison point, but two runs removes any concern about judge drift.

2. **Run the formal statistical analysis.** Pool the debate-new-moderator data (this experiment, N=8; plus whatever re-run happens) against the control-new-prompt data (the prior experiment's N=10; plus any re-run). Welch's t-test per scenario, Bonferroni-corrected over 8 tests, plus a pooled comparison. This is what the statistical-analysis skill did before and should be repeated with the post-optimization data.

3. **Treat this as strong-but-not-conclusive evidence** that the debate workflow has a real structural advantage over the optimized control. The bias delta is the most defensible claim; the RD delta is weaker and needs more data. The directional consistency across scenarios is the most important signal at this data volume.

4. **Update the README after the confirmation experiment runs.** The current README framing should be softened to reflect (a) the effect survived optimization but at smaller magnitude, and (b) the bias advantage is more robust than the RD advantage, and (c) the cost/latency overhead of 3–4× is only worth it when bias matters most — emotionally ambiguous decisions where single-agent prompting is most prone to drift.

5. **Consider investigating the socket error rate.** Nine runs in this experiment produced transient socket errors (including two that caused zero-score exclusions). If this becomes a recurring pattern, it may warrant a retry-config investigation or a migration to a more reliable API endpoint — not because it biases the data (Mastra's retries recover cleanly in most cases) but because it degrades the per-run success rate at N=10+.
